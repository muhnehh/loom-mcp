How It Works

Three simple steps to efficient code retrieval.

1

Index Once

Run loom_get_topology to index your codebase. Tree-sitter parses all source files and extracts symbol metadata. Happens once per project.

2

Search by Intent

The AI calls search_symbols(query) or loom_search_symbols. The server performs keyword + semantic search against the index in milliseconds.

3

Surgical Retrieval

Only the matching symbol's source code is returned — not the surrounding file, not unrelated classes. Byte-offset retrieval.

FAQ

How is LoomMCP different?
LoomMCP uses the same tree-sitter approach as alternatives but focuses on higher token reduction through better skeletonization. We also offer GPU embeddings via @xenova/transformers.
Which MCP clients are supported?
Claude Code, Claude Desktop, Cursor, Windsurf, Continue, VS Code with MCP extension, and any MCP-compatible client.
Why not just use larger context?
Bigger context = higher costs. Every token you don't send saves money. Plus, larger contexts can lead to worse model recall and higher latency.
How much can I reduce token usage?
Our benchmark shows 97.75% average reduction. For a 50K token project, you'd use ~1,250 tokens instead of 50,000.
